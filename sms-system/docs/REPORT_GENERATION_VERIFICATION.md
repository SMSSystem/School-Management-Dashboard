## Prerequisites

### Mock mode vs. live mode

The app has two data modes controlled by `USE_MOCK` in [`src/lib/data.ts`](../src/lib/data.ts):

| Mode | Reports list | PDF export | Generate Report button |
|---|---|---|---|
| `USE_MOCK = true` (default) | Shows 3 mock records | ✅ Works | ❌ Writes to Firestore — will fail without credentials |
| `USE_MOCK = false` | Reads from Firestore | ✅ Works | ✅ Works |

Sections 1–7 of this guide work fully in mock mode. Section 8 (Generate Report) requires live mode.

| Step | Role required |
|---|---|
| Reports list, PDF, download | Any role |
| Generate / Re-generate | `institution_admin` or `senior_teacher` |

## 3. PDF Modal — Opening and Chunk Loading

1. Sign in as any role with at least one visible report.
2. Click the **PDF** button on any row.
3. Verify the following sequence on **first click** (fresh page load):
   - A black overlay appears briefly with the text "Loading PDF renderer…" — this is the `Suspense` fallback while the `@react-pdf/renderer` chunk downloads (~1.3 MB, deferred from the initial bundle)
   - The full-screen modal replaces the overlay: toolbar at top, rendered PDF fills the remaining height
4. Close the modal and click **PDF** again. Verify the modal opens **immediately** with no loading overlay — the chunk is cached for the remainder of the session.

## 9. Generate Report (Live Mode Only)

> Requires `USE_MOCK = false` and valid `VITE_FIREBASE_*` environment variables. Firestore must contain at least one `students`, `terms`, and `institutions` document seeded for the signed-in `institutionId`.

### Initial generation

1. Sign in as `institution_admin`.
2. Click the **Generate Report** button (yellow circle icon, top-right of the page).
3. Select a student from the dropdown.
4. Select a term from the dropdown.
5. Click **Generate**.
6. Verify:
   - The button shows "Generating…" while the async operation runs.
   - The panel closes automatically on success.
   - The new report appears in the list with correct student name, term, and overall score.
7. Click **PDF** on the new report. Verify the PDF header shows the correct `institutionName`, `studentName`, and `termName` — these are denormalized from Firestore at generation time and stored on the report document, not resolved at render time.

### Re-generation (upsert)

1. Click **Re-generate** on an existing report.
2. Verify:
   - The record updates in place (overallScore, generatedAt refresh).
   - No duplicate record appears — the upsert query detects the existing document and calls `updateDoc`.

### Error handling

1. Click **Generate** without selecting a student or term.
2. Verify the button remains disabled.
3. To test a Firestore error (e.g. rules rejection), attempt generation as `regular_teacher` by temporarily modifying the `canGenerate` guard — the error message should appear below the dropdowns.

## 10. Known Limitations

| Limitation | Detail |
|---|---|
| Teacher name fallback in mock mode | Mock `feedback` entries have no `teacherName`. The PDF shows "Teacher" as a fallback. Live reports show real names. |
| Double PDF render per modal open | `PDFViewer` and `PDFDownloadLink` each render `<ReportPDF>` independently — two renders per modal open. This is a `@react-pdf/renderer` library limitation; not a bug in this implementation. No perceptible performance impact for a report of this size. |
| `PDFViewer` on mobile | The viewer renders an `<iframe>` with a blob URL. Some mobile browsers (particularly iOS Safari) block iframe blob URLs — the preview may not display. The **Download PDF** button works as a fallback on all platforms. |
| Tailwind has no effect inside `ReportPDF` | `@react-pdf/renderer` components are not DOM elements. All PDF styling uses `StyleSheet.create()`. Changes to the PDF visual style must be made there, not in Tailwind classes. |
